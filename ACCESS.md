# Как мы работаем с кодом

## Кто чем может заниматься

| Ты | Можешь изменять | Не трогай без согласования |
|----|-----------------|---------------------------|
| **Дизайнер-фронт** | `/frontend/src/components/` (стили компонентов)<br>`/frontend/public/images/` (картинки)<br>`/docs/design/` (документация по дизайну)<br>`/frontend/src/assets/styles/design-tokens.css` (цвета и шрифты) | Бэкенд<br>Инфраструктуру<br>API эндпоинты |
| **Аналитик** | `/docs/analytics/` (исследования)<br>`/docs/project/` (документация)<br>Создавать issues с багами и фичами | Код (без PR) |
| **Бэкендер** | `/backend/` (весь бэкенд)<br>`/docs/api/` (API документация) | Фронтенд (если не касается API)<br>Инфраструктуру |
| **DevOps** | Абсолютно всё | — |
| **PM (тимлид)** | Абсолютно всё | — |

## Как вносить изменения

### Если ты дизайнер и хочешь поправить стили кнопки:
1. Создай ветку `feature/fix-button-styles`
2. Найди файл `/frontend/src/components/common/Button/Button.module.css`
3. Поправь CSS
4. Создай Pull Request
5. GitHub сам позовёт фронтендера и PM на ревью
6. После апрува можно вливать

### Если ты бэкендер и добавляешь новый эндпоинт:
1. Создай ветку `feature/add-new-api`
2. Работай в `/backend/`
3. Обнови документацию в `/docs/api/`
4. Создай PR (CODEOWNERS добавит DevOps и PM)

### Если ты аналитик и хочешь добавить отчёт:
1. Работай в `/docs/analytics/`
2. Создай PR с пометкой "documentation"
3. PM проверит и вольёт

### Если нужно срочно поправить баг:
1. Создай ветку `hotfix/...`
2. Внеси изменения (даже если это не твоя зона, но ты знаешь как)
3. В PR обязательно тегни @teamup-dev/pm и того, кто отвечает за эту зону

## Важно!
- **Никто не пушит сразу в main** (ветка защищена)
- **Все изменения проходят через Pull Request**
- **PR должен получить апрув** от владельцев кода (см. CODEOWNERS)
- Если сомневаешься — создавай PR и спрашивай в комментариях
